# AI-Powered Restaurant Analytics Dashboard вҖ” Design Spec

**Date:** 2026-06-25
**Author:** Md Mozammel Hossain
**For:** Academic AI engineering project (graded against detailed rubric, target: full marks)

## 1. Purpose

Build an end-to-end system that turns historical POS sales data into actionable
foresight for restaurant operators: revenue forecasting, inventory risk detection,
and staffing recommendations, all surfaced through a multi-branch dashboard with
AI-generated natural-language insights.

This is an academic project scoped to be buildable solo in a few weeks, but
designed with production-grade engineering practices (architecture, testing,
security, documentation) to score well against the evaluation rubric, particularly
on AI integration depth, technical quality, and deployment/documentation.

## 2. Scope

Full scope, built together (not staged as MVP-then-expand):

1. CSV data ingestion with column mapping
2. Prophet-based revenue forecasting (7-day and 30-day, confidence bands, multi-branch)
3. Inventory intelligence (stockout/overstock alerts, reorder suggestions)
4. Staffing suggestions (per-shift staff counts, efficiency scoring)
5. LLM insight layer (natural-language summaries/recommendations over the above)
6. Central dashboard (KPI cards, peak-hour heatmap, branch/date filtering)

Data: synthetic, generated by us вҖ” multi-branch restaurant sales with realistic
seasonality, weekly patterns, and holiday spikes, used to train/demo Prophet and
exercise the inventory/staffing logic.

- **Frontend:** Next.js (React) + TailwindCSS + Recharts for charts/heatmaps.
- **Backend:** FastAPI (Python), layered as `api/` (routes) вҶ’ `services/`
  (business logic: forecasting, inventory, staffing, insights) вҶ’ `db/`
  (SQLAlchemy models + queries).
- **Database:** PostgreSQL. Local dev via Docker Compose; deploy target
  decided later (Neon/Supabase free tier likely).
- **Forecasting:** Facebook Prophet, one model trained per branch on
  `sales_transactions`, retrained on each new CSV upload.
- **LLM insight layer:** OpenRouter API, DeepSeek free-tier model
  (`deepseek/deepseek-chat-v3` or current free slug). Used to turn
  forecast/inventory/staffing outputs into structured natural-language
  summaries and recommendations вҖ” not a raw chat wrapper.
- **Containerization:** Docker Compose bundling frontend, backend, and db
  for reproducible local run and as a base for later deployment.

## 4. Data Model (PostgreSQL)

- `branches` вҖ” id, name, location
- `sales_transactions` вҖ” id, branch_id, timestamp, item, quantity, amount
- `inventory_items` вҖ” id, branch_id, sku, name, current_stock, reorder_threshold, unit_cost
- `forecasts` вҖ” id, branch_id, date, predicted_revenue, lower_bound, upper_bound, generated_at
- `staffing_recommendations` вҖ” id, branch_id, shift (morning/afternoon/evening), date, recommended_staff_count, efficiency_score
- `csv_uploads` вҖ” id, branch_id, filename, column_mapping (JSON), status, uploaded_at
- `ai_insights` вҖ” id, branch_id, type (weekly_summary/anomaly/recommendation), content, generated_at
- `users` вҖ” id, email, hashed_password (single-tenant auth, restaurant owner/admin)

## 5. Core Modules

### 5.1 Data Ingestion
- Drag-and-drop CSV upload (frontend) вҶ’ backend validates file type/size/row count
  вҶ’ column mapping step (user maps their CSV columns to our schema: date, item,
  quantity, amount, branch) вҶ’ parser writes to `sales_transactions`.

### 5.2 Forecasting Engine (Prophet)
- One Prophet model per branch, trained on daily aggregated revenue.
- Outputs: 7-day and 30-day forecast with yhat/yhat_lower/yhat_upper.
- Detects weekly seasonality and trend automatically (Prophet built-in);
  holiday effects modeled via a custom holidays list seeded into the
  synthetic data generator and passed to Prophet.
- Evaluation: MAE/RMSE computed on a held-out tail of each branch's history,
  surfaced in the dashboard and documented in README.

### 5.3 Inventory Intelligence
- Rule-based + forecast-driven: expected demand per SKU derived from sales
  velocity (recent consumption rate) combined with the revenue forecast trend.
- Stockout alert: `days_to_run_out = current_stock / avg_daily_consumption`,
  flagged if below a configurable threshold.
- Overstock flag: stock significantly exceeding N days of average consumption.
- Reorder suggestion: target stock level minus current stock, bounded by
  lead time assumptions.

### 5.4 Staffing Suggestions
- Per-shift (morning/afternoon/evening) recommended staff count derived from
  forecasted order volume for that time window, using a configurable
  orders-per-staff ratio.
- Efficiency score: actual staffing (if provided) vs. recommended, surfaced
  as a percentage.
- Weekend/holiday buffer: multiplier applied on detected high-traffic days
  (from Prophet's seasonality components).

### 5.5 LLM Insight Layer
- Backend service collects: latest forecast deltas, active inventory alerts,
  staffing recommendations for the period.
- Sends a structured prompt (system prompt fixes role + output schema) to
  DeepSeek via OpenRouter, requesting JSON output: `{summary, key_risks,
  recommendations[]}`.
- Response validated against a Pydantic schema before storage/display;
  invalid/malformed responses are retried once, then fall back to a
  rule-based templated summary (graceful degradation, no LLM single point
  of failure for core dashboard function).
- Insight generation is rate-limited per branch (e.g., max once per hour)
  to respect the free-tier quota and prevent abuse.

### 5.6 Dashboard
- KPI cards: revenue, order volume, average order value (branch + date
  range filtered).
- Peak-hour heatmap: day-of-week x hour-of-day revenue/order density.
- Forecast chart: historical + forecast line with shaded confidence band.
- Inventory panel: stockout/overstock alert list.
- Staffing panel: shift planner view with recommended vs actual.
- Insight panel: latest LLM-generated summary and recommendations.

## 6. Safety & Guardrails

- **Input validation:** CSV file type/size/row-count limits; column mapping
  validated before parse; reject malformed rows with a clear per-row error
  report instead of silently dropping or crashing.
- **Prompt injection resistance:** system prompt fixes the LLM's role and
  output schema; user-derived data is passed as labeled data fields, never
  concatenated as instructions.
- **Output handling:** LLM output is parsed as JSON against a Pydantic
  schema before being trusted; rendered in the frontend as plain text/structured
  fields, never as raw HTML (XSS prevention).
- **Rate limiting:** insight-generation endpoint capped per branch per hour.
- **Auth:** JWT-protected API routes; passwords hashed with bcrypt.
- **Secrets:** all keys (OpenRouter, DB credentials) in `.env`, excluded via
  `.gitignore`; never committed or hardcoded.
- **Audit log:** `ai_insights` table doubles as an audit trail (timestamp,
  branch, type, content) for every AI generation.

## 7. Testing

- **Backend (pytest):** unit tests for Prophet wrapper (forecast shape/bounds),
  inventory reorder logic, staffing calculation, LLM JSON-schema validation
  and fallback path.
- **Integration test:** CSV upload вҶ’ parse вҶ’ forecast generation, end-to-end
  against a test database.
- **Frontend:** component tests (Vitest + React Testing Library) for KPI
  cards, forecast chart, and insight panel rendering states (loading/error/empty/success).

## 8. Deployment

Decided later, once core build is stable. Candidate plan: Dockerized
frontend + backend, Postgres on a free-tier managed provider (Neon/Supabase),
frontend on Vercel, backend on Render/Railway. Docker Compose used for local
dev regardless of final deploy target, so the project is deploy-ready early.

## 9. Out of Scope (for this iteration)

- Multi-tenant SaaS auth (multiple restaurant *companies* on one deployment) вҖ”
  single-tenant, multi-branch only.
- Real POS API integrations вҖ” CSV upload only.
- Mobile app вҖ” responsive web dashboard only.
